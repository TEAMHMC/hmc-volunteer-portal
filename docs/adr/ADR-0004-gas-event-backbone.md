# ADR-0004: Google Apps Script as Event Management Backbone

## Status

Accepted

## Context

Before the volunteer portal existed, HMC managed events entirely in Google Sheets: event
records, RSVP lists, attendance, and outbound email confirmations all ran through Sheets
and Google Apps Script (GAS). This system was built by the operations team, is actively
maintained, and runs on existing Google Workspace licenses at zero incremental cost.

When the portal was introduced, a decision was needed on whether to migrate event management
fully into Cloud Run and Firestore, retain GAS, or adopt a third-party event platform.

Key constraints:

- The operations team does not have engineering capacity for a full migration.
- Google Sheets remains the source of truth for event scheduling, used in daily workflows
  by non-technical staff.
- RSVP confirmation emails, reminder emails, and attendance sheets are all currently
  generated by GAS with no incidents in over a year.
- Budget does not support a paid event platform.

## Decision

Retain **Google Apps Script** as the event backbone for event creation, RSVP ingestion,
confirmation email delivery, and attendance tracking. Firestore is used only for
volunteer/partner profile data.

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| Migrate to Cloud Run + Firestore + SendGrid | Full programmatic control; TypeScript; testable; unified data store | High migration effort; ops team loses Sheets workflow; email deliverability setup; ongoing cost |
| Retain GAS (current approach) | Zero cost; Sheets-native; team already knows it; no migration risk | No TypeScript; GAS deployments are manual; two data stores to sync; postMessage bridge needed for frontend embedding |
| Third-party event platform (e.g., Eventbrite, Luma) | Polished UX; built-in RSVP and email | Per-event fees or subscription cost; data lives outside HMC systems; limited customization for community health context |

## Decision Outcome

GAS is retained as the event backbone. The portal frontend communicates with GAS via
`postMessage` through an iframe bridge for embedded RSVP flows, and via direct `doPost`
webhook calls from Cloud Run for server-side event sync.

Firestore stores volunteer and partner data; GAS owns event and RSVP data in Sheets.
A lightweight sync layer (GAS writes a Firestore document on RSVP confirmation) keeps the
two stores loosely consistent without requiring a full migration.

## Consequences

**Positive**

- Zero additional cost; GAS runs on existing Google Workspace.
- Operations team retains full visibility into event data via familiar Sheets UI.
- Email deliverability via Google's SMTP infrastructure is well-established.
- No migration risk or downtime during portal rollout.

**Negative**

- Two data stores must be kept in sync; any inconsistency between Sheets and Firestore
  requires manual reconciliation.
- GAS deployments require a human to publish a new version through the Apps Script IDE;
  there is no automated CI/CD pipeline for GAS code.
- No TypeScript, no unit test framework, and limited version control visibility compared to
  the GitHub-hosted portal codebase.
- The postMessage bridge between the portal frontend and the GAS-backed iframe adds
  complexity to RSVP flows and requires careful origin validation to prevent spoofing.

**Neutral**

- The GAS web app URL is stored as an environment variable in Cloud Run and the portal
  frontend config; rotating the URL (after a GAS redeploy) requires updating both locations.
- Eventbrite integration for external event discovery is handled by a separate GAS function
  that deduplicates by Eventbrite ID before writing to the Sheets backbone.
- Future migration to a unified Cloud Run event service remains an option as team capacity
  grows; the current sync layer is designed to be removable.
